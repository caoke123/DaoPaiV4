# DaoPai V3 总体架构设计

> 版本：v1.0（Phase 1 架构冻结草案）
> 适用范围：DaoPai V3 SaaS 方向
> 关联文档：V3_DATA_MODEL.md / V3_AGENT_DESIGN.md / V3_CLOUD_PLATFORM.md / V3_ROADMAP.md

---

## 1. V3 定位

DaoPai V3 是面向多网点使用的 SaaS 系统，整体形态为：

- **云端（DaoPai Cloud Platform）**：负责管理、数据、用户、租户、任务中心、日志中心。
- **本地（DaoPai Local Agent）**：负责员工电脑上的浏览器自动化执行。
- **明确边界**：云端不直接控制本地浏览器，所有浏览器执行动作发生在 Local Agent 所在的员工电脑上。

V3 基于 V2 已稳定的执行能力初始化而来，但 V3 是全新的 SaaS 方向，不再沿用 V2 单机/单租户的运行假设。V2 仍是当前稳定生产工具，V3 开发不得破坏 V2 运行链路。

---

## 2. 为什么拆成 Cloud Platform + Local Agent

DaoPai 不是普通网页系统，核心任务必须在本地电脑执行浏览器自动化（Playwright / 已有窗口复用）。直接把浏览器自动化搬到云端会带来以下问题：

- 云服务器无法稳定访问员工电脑本地的浏览器实例和已登录会话。
- 把浏览器执行搬到云端会丢失员工电脑本地已登录的笨鸟业务会话、浏览器用户数据和网点业务账号会话。
- 云服务器让员工电脑浏览器端口直接对外暴露，存在严重安全风险。
- 一旦员工关机、网络抖动，云端直接控制会立即失败且难以恢复。

因此 V3 采用：

- **Local Agent 主动连接云端**：由本地主动拉取任务、主动上报心跳与结果。
- 云端只下发任务和数据，不直接发起浏览器操作。
- 本地 Agent 失联时云端仅记录离线状态，不阻塞其它设备任务。

这种模型更安全、更稳定，也更符合员工电脑不可 7x24 在线的现实。

## 4. Cloud Platform 职责

云端平台承担所有"管理类"职责：

- **登录与会话**：用户登录、Token 颁发、会话保持。
- **租户管理**：租户开通、停用、到期、最大设备数限制。
- **网点管理**：租户下的网点（site）增删改查。
- **用户与授权**：租户内用户、角色、简单授权（第一版不做复杂 RBAC）。
- **任务中心**：任务创建、状态机、分配、领取记录、结果归档。
- **日志中心**：Agent 上报的执行日志、进度、错误聚合。
- **数据隔离**：所有业务查询强制按 `tenant_id` 隔离。
- **云端 PostgreSQL**：业务数据唯一持久化目标。
- **设备绑定**：workstation 注册、绑定、心跳状态、授权状态。

云端不做：本地浏览器执行、本地窗口状态判定、本地截图存储。

---

## 5. Local Agent 职责

Local Agent 安装在每台执行任务的员工电脑上，只负责"当前这台电脑"的执行能力：

- **本地环境检测**：Node 版本、依赖、Playwright 浏览器是否就绪。
- **浏览器环境检测**：目标浏览器是否已安装、是否可启动、已登录会话是否可用。
- **本地任务执行**：从云端领取任务后，调用本地 Playwright / 浏览器自动化执行。
- **窗口状态上报**：本机 READY / BUSY / ERROR 等状态只代表本机。
- **任务拉取**：通过 HTTP 轮询云端获取待执行任务。
- **日志 / 进度 / 结果回传**：执行过程阶段性上报，最终回传 waybill 结果。
- **本地临时日志**：本地保留最近 N 天日志，便于现场排查。
- **可选本地截图**：调试模式下本地保存截图，不默认上传云端。

Agent 不负责：租户管理、SaaS 会员授权管理、云端任务中心 UI、跨设备调度。

## 6. 通信方式

第一版通信策略：

- **HTTP API + 轮询**：Agent 周期性向云端拉取待执行任务、上报心跳与结果。
- **轮询间隔**：默认 5～10 秒，可配置；任务进行中可加快进度上报频率。
- **任务可靠性靠 PostgreSQL 状态机**：任务状态、领取锁、完成时间全部以数据库为准，HTTP 仅作为传输层。
- **WebSocket 后续增强**：后续可引入 WebSocket 做 `task_available` 实时通知，但：
  - WebSocket 不能替代数据库任务状态机。
  - WebSocket 不能替代 HTTP 任务详情拉取。
  - Agent 收到 WebSocket 通知后，仍通过 HTTP 拉取完整任务详情。
  - 断线后必须回退到轮询兜底，不能因 WebSocket 断开而漏任务。

## 7. 截图策略

截图策略遵循"默认不存、本地临时、云端只存地址"：

- **生产默认关闭截图**：避免占用云服务器磁盘和带宽。
- **不把截图存进轻量云服务器 Docker**：避免磁盘被打满影响业务。
- **调试时本地临时保存**：Agent 可在 `screenshots/` 目录临时保存，调试结束后清理。
- **未来远程排查需求**：接入对象存储（S3 / R2 / OSS），云端只保存对象地址。
- **云端数据库最多保存截图地址**：不保存图片二进制。

## 8. 当前不做什么

V3 第一版明确不做以下事项，避免过度设计：

- 不做纯云端浏览器自动化（执行必须在员工本地电脑）。
- 不让云端直接连接本地浏览器端口。
- 不做复杂支付系统（第一版手动开通/停用/到期控制）。
- 不做复杂权限系统（第一版简单角色 + 租户隔离即可）。
- 不做纯 WebSocket 任务系统（任务可靠性靠数据库状态机 + HTTP 拉取）。
- 不把截图存进云服务器 Docker。
- 不破坏 V2 稳定执行链路（V3 开发期间 V2 继续生产）。
- 不引入 Kubernetes / 微服务 / 复杂消息队列（第一版单体 Cloud API 足够）。

---

## 9. 与 V2 的关系

- V2 是当前稳定生产工具，V3 开发期间 V2 必须可继续运行。
- V3 不修改 V2 任何代码与配置。
- V3 复用 V2 已稳定的浏览器自动化执行能力作为 Local Agent 的执行内核基础。
- V3 不把 V2 历史问题当作 V3 需求，V3 是全新 SaaS 设计。
- V3 与 V2 端口、数据库、Redis 完全隔离（V3：5176 / 3300 / 5436 / daopai_v3 / 6381）。
