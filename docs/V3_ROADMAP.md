# DaoPai V3 开发路线图

> 版本：v1.0（Phase 1 架构冻结草案）
> 适用范围：DaoPai V3 SaaS 方向
> 关联文档：V3_ARCHITECTURE.md / V3_DATA_MODEL.md / V3_AGENT_DESIGN.md / V3_CLOUD_PLATFORM.md

---

## 总体原则

- 每个 Phase 必须有明确目标、修改范围、禁止事项、验收标准、风险点。
- 任何 Phase 都不得破坏 V2 稳定运行。
- 任何 Phase 都不得跳过数据隔离原则。
- 任何 Phase 都不得引入未在架构文档中约定的新依赖（K8s / 微服务 / 消息队列 / 支付等）。

---

## Phase 1：架构文档冻结

### 目标

完成 V3 SaaS 架构设计文档，作为后续所有开发阶段的依据。本阶段只写文档，不写代码。

### 修改范围

- 新增 `docs/V3_ARCHITECTURE.md`
- 新增 `docs/V3_DATA_MODEL.md`
- 新增 `docs/V3_AGENT_DESIGN.md`
- 新增 `docs/V3_CLOUD_PLATFORM.md`
- 新增 `docs/V3_ROADMAP.md`

### 禁止事项

- 不修改任何业务代码、配置、依赖。
- 不开始 tenantId / workstationId 代码改造。
- 不触碰 V2 目录。

### 验收标准

- 5 份文档全部存在且内容完整。
- 文档覆盖架构、数据模型、Agent、云端、路线图五大维度。
- 文档已 git 提交。
- 后续 Phase 可基于此文档独立展开。

### 风险点

- 文档过度设计导致后续难以落地 → 第一版以简单可靠为主。
- 文档遗漏关键决策 → 进入 Phase 2 前由用户确认（见完成报告第 6 节）。

## Phase 2：多租户数据模型

### 目标

在 V3 后端落地 `tenantId` / `siteId` / `workstationId` 三元组数据模型，建立数据隔离基础。

### 修改范围

- 数据库迁移脚本（新增表结构）。
- 后端数据访问层（注入 tenant_id 过滤）。
- 后端 API 中间件（从会话提取 tenant_id）。

### 禁止事项

- 大改业务执行链路。
- 修改 BrowserPool / 窗口状态 / 任务执行内核。
- 修改 settings.json 结构。
- 触碰 V2 目录。

### 验收标准

- 核心表（tenants / users / sites / employees / workstations / tasks / task_logs / waybill_results / agent_heartbeats）已建表。
- 所有业务查询强制带 tenant_id。
- 跨租户访问返回 403。
- 单元测试覆盖隔离逻辑。

### 风险点

- 遗漏某条查询未加 tenant_id → 数据泄露风险，需代码审查 + 自动化检测。
- 迁移脚本与现有数据冲突 → 先在空库演练。

---

## Phase 3：Cloud Platform 基础

### 目标

搭建云端 API 基础框架：登录、租户上下文、统一错误处理、API 路由分层。

### 修改范围

- 后端入口、路由分层、错误处理中间件。
- 登录与会话模块（JWT 或 Session + Redis）。
- 租户上下文中间件。

### 禁止事项

- 不引入复杂 RBAC。
- 不引入支付系统。
- 不修改 V2 任何代码。
- 不破坏 Phase 0 已通过的启动链路。

### 验收标准

- 登录 / 登出 / 会话保持可用。
- 所有 API 经统一错误中间件，未知路径立即 404。
- 租户上下文注入所有业务路由。
- 前端 5176 / 后端 3300 启动正常。

### 风险点

- 会话设计不当导致安全问题 → Token 短期 + Refresh Token。
- 错误处理遗漏导致进程崩溃 → 全局兜底中间件 + 单元测试。

---

## Phase 4：租户 / 用户 / 授权

### 目标

实现租户与用户管理：active / disabled / expired 状态、expires_at、max_workstations 控制。

### 修改范围

- 租户管理 API 与后台页面。
- 用户管理 API 与后台页面。
- workstation 数量限制逻辑。

### 禁止事项

- 第一版不做支付。
- 不做复杂角色权限树。
- 不修改 V2 目录。

### 验收标准

- 超级管理员可开通 / 停用 / 续期租户。
- 到期租户用户无法登录、Agent 无法鉴权。
- 超过 max_workstations 时拒绝新增设备。
- 操作日志可审计。

### 风险点

- 到期判定时区错误 → 统一使用 UTC 存储，前端展示本地时区。
- 手动操作误删租户 → 软删除 + 操作日志 + 二次确认。

---

## Phase 5：workstation 设备绑定

### 目标

实现本地电脑注册、绑定、心跳、在线状态管理。

### 修改范围

- workstation 注册 / 绑定 API。
- 心跳上报 API。
- 在线状态判定定时任务。

### 禁止事项

- 不修改 BrowserPool 启动逻辑（参考 Phase 0-C ISSUE-002，留到 Agent 拆分阶段）。
- 不修改 V2 设备相关代码。
- 不让云端直接连接本地浏览器。

### 验收标准

- Agent 可使用 agentToken 完成首次注册。
- 心跳上报正常，云端可查询在线状态。
- 超时未心跳的 workstation 自动标记 offline。
- offline 状态下不分配新任务。

### 风险点

- 心跳频率过高导致数据库压力 → Redis 缓存最近心跳，定时批量落库。
- 多台电脑使用同一 workstationId → 第一版以 Token 撤销 + 日志告警为主。

## Phase 6：Local Agent 基础拆分

### 目标

将 V2 浏览器自动化执行能力抽取为独立的 Local Agent 程序：启动、鉴权、环境检测、心跳上报。

### 修改范围

- **第一版 Agent 建议放在 DaoPaiV3 仓库内的独立子包/子目录**，例如 `packages/agent`，便于共享任务类型、API 协议和状态枚举。后续稳定后再评估是否拆分为独立仓库。
- Agent 启动入口、配置加载、鉴权模块、心跳模块。
- Agent 本地环境与浏览器检测逻辑。

**推荐项目结构**（仅路线建议，本任务不要求创建这些目录，也不调整现有项目结构）：

```text
DaoPaiV3/
  └─ packages/
      ├─ cloud/         云端平台（cloud-web + cloud-api）
      ├─ agent/         本地执行端（Local Agent）
      └─ shared/        Cloud 与 Agent 共享类型、任务状态、API 协议、状态枚举
```

说明：

- `packages/shared` 用于共享任务状态机、workstation 状态枚举、API 请求/响应类型，避免 Cloud 与 Agent 协议不一致。
- 当前只是路线建议，**不要求本任务创建这些目录**。
- **不要在本任务中真正调整项目结构**。

### 禁止事项

- 先不执行真实任务（仅完成启动 + 鉴权 + 心跳）。
- 不修改 V2 执行内核代码（抽取而非搬运）。
- 不破坏 V2 运行链路。

### 验收标准

- Agent 可独立启动并读取 `agent.json`。
- Agent 可完成云端鉴权与心跳上报。
- Agent 可检测本地浏览器环境并上报状态。
- V2 仍可独立运行，不受影响。

### 风险点

- 抽取边界不清导致 V2 被破坏 → 优先复制后重构，保持 V2 不动。
- Agent 与 Cloud 协议不一致 → 先冻结协议文档再开发。

## Phase 7：任务下发和结果回传

### 目标

打通完整任务链路：云端创建任务 → Agent 轮询领取 → 本地执行 → 回传日志/进度/结果。

### 修改范围

- 云端任务创建 API、领取 API、结果回传 API。
- Agent 任务轮询、领取、执行、回传模块。
- 任务状态机实现。

### 禁止事项

- 不引入 WebSocket（第一版只用 HTTP 轮询）。
- 不修改 V2 已稳定的浏览器操作内核。
- 不把截图存进云服务器 Docker。

### 验收标准

- 云端可创建任务并指定 tenant / site / workstation。
- Agent 可领取任务并执行真实浏览器自动化。
- 执行日志和结果正确回传云端。
- 任务状态机正确流转（pending → assigned → running → succeeded/failed）。
- 多设备并发领取不冲突。

### 风险点

- 任务重复执行 → 数据库原子领取锁 + 幂等回传。
- 任务丢失 → 状态超时回收机制 + 失败重试策略。
- Agent 崩溃导致任务卡住 → 超时检测 + 人工回收后台。

## Phase 8：截图策略重构

### 目标

落地截图策略：默认关闭、本地临时保存、可选对象存储（S3 / R2）。

### 修改范围

- Agent 截图模块（本地保存逻辑）。
- 云端对象存储配置（可选，第一版可不接入）。
- 云端 waybill_results 表的 `screenshot_url` 字段。

### 禁止事项

- 不默认开启截图。
- 不把截图二进制存进 PostgreSQL。
- 不把截图存进云服务器本地磁盘。

### 验收标准

- 生产模式默认不截图。
- 调试模式可本地临时保存，任务完成后自动清理。
- 配置对象存储后，Agent 可直传并回传地址。
- 云端数据库只保存截图 URL。

### 风险点

- 对象存储凭证泄露 → 使用临时 STS Token，不长期保存 AccessKey。
- 截图占用本地磁盘 → 自动清理 + 容量限制。

## Phase 9：生产部署和新电脑初始化

### 目标

完成云端生产部署（Docker / Nginx / HTTPS / 备份）和 Agent 打包分发流程。

### 修改范围

- 云端 `docker-compose.yml` 生产版本。
- Nginx 反向代理与 HTTPS 配置。
- PostgreSQL 定时备份脚本。
- Agent 打包脚本（Windows 可执行 / 安装包）。
- 新电脑初始化文档（浏览器环境、Agent 安装、配置下发）。

**新电脑初始化浏览器运行环境**：

新电脑初始化阶段必须处理浏览器运行环境问题，优先评估以下方案，避免依赖员工电脑已安装 Chrome 和现场外网下载：

- **固定浏览器包**：随 Agent 打包 Chrome for Testing 或固定版本 browser runtime，解压即用。
- **离线安装包**：Agent 安装包内置浏览器二进制，无需现场联网下载。
- **版本固定**：浏览器版本与 Agent 版本绑定，避免 Chrome 自动更新导致兼容性问题。

新电脑初始化文档必须包含：

- 浏览器运行环境检测步骤。
- 固定浏览器包安装/解压步骤。
- Agent 安装与 `agent.json` 配置下发步骤。
- 首次启动与云端注册验证步骤。

### 禁止事项

- 不引入 Kubernetes（第一版单体 Docker 足够）。
- 不引入复杂 CI/CD 流水线（第一版手动部署 + 脚本即可）。
- 不破坏 V2 生产环境。

### 验收标准

- 云端可通过 HTTPS 访问，证书有效。
- PostgreSQL 每日备份正常执行。
- Agent 可在新电脑上完成安装、配置、首次注册。
- 新电脑浏览器环境初始化有清晰文档。
- 故障恢复流程有文档说明。

### 风险点

- HTTPS 证书过期 → 使用 certbot 自动续期。
- 备份未验证可恢复性 → 每月恢复演练。
- 新电脑初始化步骤复杂 → 提供一键脚本 + 文档。

## 跨 Phase 通用约束

- 任何 Phase 不得破坏 V2 运行。
- 任何 Phase 不得跳过数据隔离原则。
- 任何 Phase 不得引入未约定的新依赖（K8s / 微服务 / 消息队列 / 支付 / 复杂 RBAC）。
- 任何 Phase 完成后必须更新 `docs/V3_PHASE0_KNOWN_ISSUES.md` 中相关 ISSUE 的处理状态。
- 任何 Phase 进入开发前必须由用户确认架构决策。
