### 1.3 上一站选择对比【重点】

#### V2 方案（`selectPrevStation`）

```
Step 1: page.click('.arrivalscan_left .el-input--suffix input')
Step 2: page.waitForTimeout(800)  ← 固定等待
Step 3: page.evaluate(el.click()) 遍历 li.el-select-dropdown__item，
        文本匹配 "天津分拨中心"，DOM click
Step 4: 若 Step 3 失败 → page.fill(input, "天津分拨中心") + Enter 兜底

Step 6: 若 fill 兜底 → 读取 input.value 验证

#### V3 方案（`stableFillPrevStation`）

```
Step 1: findPrevStationInputByLabel（遍历 .arrivalscan_left input，
        向上找行容器，检查 textContent 含"上一站"且不含"班次"）
Step 2: assertNotShiftField（再次校验不在班次行）
Step 3: clickPrevStationTrigger（依次尝试 input/suffix/wrapper 点击，
        每个尝试后 waitForFunction 检查 popper 是否出现含目标文本）
Step 4: clickVisiblePrevStationOption（evaluate 获取 popper 中可见候选项坐标，
        page.mouse.click 坐标点击；失败则 page.$$ 遍历 option handles click）
Step 5: verifyPrevStationSelected（三重校验：input.value / el-tag / li.selected）
Step 6: MAX_RETRIES=3，每次失败 Escape + waitForTimeout(300)

#### 差异分析

| 方面 | V2 | V3 | 推荐 |
|------|-----|-----|------|
| Input 定位 | `.arrivalscan_left .el-input--suffix input` 直接用（风险：可能命中班次） | label 文本向上查找 + 班次保护 + nth-child(7) 兜底 | **V3 更好**，防班次误选 |
| Popper 等待 | 固定 800ms | waitForFunction 检查 popper rect/display/visibility | **V3 更好**，条件等待更可靠 |
| 候选项点击 | `page.evaluate(el.click())` — DOM 原生点击 | 坐标点击 → handle click — Playwright 点击 | **V2 更简单可靠**（见下文分析） |
| 兜底方案 | fill + Enter（readonly input，但实际有效） | 无（审计指令要求移除） | V2 兜底值得考虑 |
| 反向校验 | 仅 fill 兜底后有校验 | 三重校验（value/tag/selected） | **V3 更好** |
| 重试机制 | 无（单次执行） | MAX_RETRIES=3 | **V3 更好** |

#### V2 `evaluate(el.click())` 为何有效

V2 使用 `page.evaluate` 遍历 DOM 中的 `li.el-select-dropdown__item`，找到包含"天津分拨中心"的选项后，调用原生 `(item as HTMLElement).click()`。

**关键发现**：V2 的 DOM click 能触发 Element UI 的 Vue 事件，因为 Element UI 的 el-select 组件对 dropdown item 的 click 事件是通过原生 DOM 事件委托绑定的，`el.click()` 可以冒泡到组件的事件处理器。

这与 V3 中 sign pageSize 的 evaluate click 失败不同——后者失败是因为 element plus 的 el-pagination 组件对 pageSize 变化通过 v-model 双向绑定监听，而 v-model 需要 input 事件触发，单纯的 DOM click 不触发 input 事件。

#### 推荐方案

| 推荐做法 | 来源 |
|---------|------|
| Input 定位 | **保留 V3** 的 label 文本定位 + 班次保护 |
| 下拉等待 | **保留 V3** 的 condition-based waitForFunction（检查 popper 可见性） |
| 候选项点击 | **采用 V2 策略**：先 `page.evaluate` DOM click（简单可靠），失败后 Playwright handle click |
| 兜底方案 | **考虑恢复 V2** 的 fill + Enter 兜底（虽 input 为 readonly，但在某些场景下有效） |
| 反向校验 | **保留 V3** 的三重校验 |
| 重试 | **保留 V3** 的 MAX_RETRIES=3 |

#### V3 不稳定的根因判断

3. **"使用"按钮点击后 Vue 未响应** → 已使用 Playwright 真实 click
4. **弹窗关闭动画未完成** → 已加 hidden wait + 兜底判断

#### V3 日期不稳定的根因分析

V3 Fix-B 的 `setSignDateRange()` 直接 fill 页面上的日期范围 input（`.search-wrap .inputs .el-date-editor input`），然后 blur + Enter。这个方案存在以下风险：

1. **Element UI 日期范围选择器的值并非直接通过 input.value 驱动**。页面上的两个 input 只是展示用，真正的数据状态在 Vue 组件内部。直接修改 input.value 再 blur/Enter，Element UI 可能不会正确解析。

2. **V2 的方案更可靠**：点击日期 input 打开面板 → 在面板内部的 input 中 fill → 点击面板的"确定"按钮 → Element UI 内部状态机处理值更新 → 关闭面板并回填页面 input。

3. **V3 没有点击"确定"按钮**，依赖 Enter 键关闭面板，但如果焦点不在面板上，Enter 可能无效。

#### 推荐方案

**强烈建议 V3 迁移到 V2 的日期选择策略**：

```
推荐执行链路（来自 V2）:
1. 点击 SIGN_SELECTORS.dateRangeInput（打开日期面板）
2. waitForSelector('.el-date-range-picker.has-time', { state: 'visible' })
3. 定位 datePickerStartInput（面板内 placeholder="开始日期"）
4. fill(startDate)  ← MM-DD 格式即可
5. 定位 datePickerEndInput（面板内 placeholder="结束日期"）
6. fill(endDate)
7. 点击 datePickerConfirm（.el-picker-panel__footer .el-button--default）
8. waitForFunction 验证 dateRangeInput[0].value 和 dateRangeInput[1].value
9. MAX_RETRIES=3，每次失败 Escape + dismissGuardingPopups

默认规则:
- 未传日期 → 当天 00:00:00 ~ 23:59:59
- 推荐格式: MM-DD HH:mm:ss 或完整 YYYY-MM-DD HH:mm:ss
```

### 2.4 条数/页选择对比【特别重点】

#### V2 方案（`PaginationAdapter.setPageSize()`）

```typescript
// Step 1: 确保无残留下拉

await page_wait(300)

// Step 2: 点击分页大小下拉

    '.el-select-dropdown:not([style*="display: none"]) .el-select-dropdown__item'
  );
  for (const item of items) {
    if ((item.textContent ?? '').trim() === text) {
      (item as HTMLElement).click();
      return true;
    }
  }
  return false;
}, '100条/页');  // 精确文本匹配

// Step 4: 等待分页重新加载

// Step 5: 验证

  const input = document.querySelector('.el-pagination .el-pagination__sizes .el-input__inner');

// Step 6: MAX_RETRIES=3，失败时 Escape + 等待
```

#### V3 方案（Fix-B `setPageSize()`）

```typescript
// Step 1: 双 Escape 清残留
await page.keyboard.press('Escape')
await page.keyboard.press('Escape')

// Step 2: 遍历所有 pageSizeInput 找正确的那一个
//         过滤条件：包含"条/页"或包含已知 pageSize 数字
for (let i = 0; i < inputCount; i++) {
  const value = await loc.inputValue()
  if (value.includes('条/页') || value.includes('10') ...) {
    inputIndex = i; break;
  }
}

// Step 3: 点击 wrapper（el-select 容器）或 input
await wrapper.click() || await input.click()

// Step 4: waitForFunction 等待含"条/页"的 popper 出现

// Step 5: 遍历所有可见 option，文本包含 pageSize 和 "条/页" → Playwright click

// Step 6: waitForFunction 验证 input.value 包含 pageSize

// Step 7: 读取 inputValue，includes(pageSize) 确认
```

#### 差异分析

| 方面 | V2 | V3 | 推荐 |
|------|-----|-----|------|
| Input 定位 | 直接 click selector | 遍历找正确 input + wrapper click | **V3 更安全**（避免点错 input，Audit-B 发现同一 popper 中混杂派件员选项） |
| 选项点击 | **evaluate DOM click**，限定 `:not([style*="display: none"])` | Playwright locator.click | **V2 更可靠**（见下文） |
| "条/页"过滤 | 不特别过滤（但精确文本匹配 `"100条/页"` 天然过滤） | 显式 waitForFunction 等含"条/页"的 popper | **V3 更安全**（Audit-B 发现 pageSize 和派件员共享 popper） |
| 文本匹配 | 精确相等 `=== '100条/页'` | `includes(pageSize) && includes('条/页')` | V2 更精确 |
| 反向校验 | evaluate 读 input.value | Playwright inputValue() | 等价 |
| 重试 | MAX_RETRIES=3 + Escape | 无 | V2 更可靠 |

#### V2 用 evaluate click 而 V3 用 Playwright click 的关键原因

PageSize 选择器是 **el-pagination 组件**的一个 el-select 子组件。这个子组件监听的是 Vue 的 `@change` 事件，而 Vue 的 v-model 双向绑定依赖 input 事件。

**V2 的 evaluate click 有效**，因为：
1. 它限定 `.el-select-dropdown:not([style*="display: none"])`，只操作当前可见的 popper
2. `el.click()` 触发原生 DOM 事件，Element UI 内部处理了选项点击 → 更新 v-model → 触发 change 事件

**V3 Fix-B 的 Playwright click** 也应该是有效的，因为 Audit-B 确认 pageSize 的 option 不在 readonly 状态下。

但 V3 之前版本用过 evaluate click 且 **无效**——原因是 Element UI 的 pageSize select 对 change 事件的监听需要完整的合成事件，而简单的 `el.click()` 在某些 Element UI 版本中不够触发完整的 v-model 更新链。

**结论**：两种方案在理论上都可行，但 V2 的 evaluate 方案在后台标签页场景下更可靠（绕过 CSS 动画检查）。

#### 推荐方案

```
推荐迁移 V2 策略，并保留 V3 的安全增强：

1. 保留 V3 的双 Escape + 多 input 遍历 + "条/页" popper 过滤（安全增强）
2. 选项点击采用 V2 策略：
   - page.evaluate 遍历 .el-select-dropdown:not([style*="display: none"]) 
     .el-select-dropdown__item
   - 精确文本匹配 '${pageSize}条/页' → el.click()
3. 保留 V3 的 waitForFunction 反向校验（验证 input.value 含 pageSize）
4. 添加 V2 的 MAX_RETRIES=3 + Escape 重试机制
5. 等待分页重新加载: waitForTimeout(1500) + waitForLoadingHidden

不同 pageSize 不需要特殊处理，统一流程：
  30条/页 → 同上流程
  50条/页 → 同上流程
  100条/页 → 同上流程（默认推荐）
  200条/页 → 同上流程
```

## 三、弹窗清理方案调查

### 3.1 V2 弹窗清理体系

```
1. PopupManager（全局单例）
   - page.on('dialog') 注册拦截器（alert accept, confirm/prompt dismiss）
   - dismissAll(page, { timeout, maxRounds, verifyAfter })
     - 多轮循环清理 el-dialog / pay-dialog / el-message-box
     - clickSmartButton: 优先级 取消 > 关闭/否/暂不 > 关闭确认的确定 > 兜底确定
     - Escape + overlay 兜底
     - URL 跳转检测（停止清理）
   - ensureClean(page): 验证无可见弹窗
   - inspect(page): 列出所有可见弹窗及文本
   - backgroundCleanup(page): 轻量后台清理

2. PopupDismiss（独立模块）
   - dismissPopups(page): 5 轮循环清除（与 PopupManager 功能重叠但独立）
   - hasVisiblePopups(page)

3. V2 中"网点余额低于警戒金额"处理
   - 注册的 dialog handler 会自动 accept alert
   - DOM 弹窗（充值弹窗）由 dismissAll 自动清理
```

### 3.2 V3 弹窗清理体系

```
1. afterPageChangedCleanup(page, log, meta, source)
   - registerNativeAlertGuard: page.on('dialog') accept alert/dismiss confirm
   - drainNativeAlerts: 循环 drain 原生 alert
   - cleanDomPopups: 移除 pay-dialog/v-modal/el-message-box 等 DOM 弹窗

2. AgentBusinessRuntime
   - ensureCleanHome / restoreCleanHome
   - 统一入口，各 DryRun 模块调用
```

### 3.3 对比

| 方面 | V2 | V3 | 建议 |
|------|-----|-----|------|
| Alert 处理 | accept + dismiss | accept + dismiss | **一致** |
| DOM 弹窗 | 多轮循环 + smart button + overlay | cleanDomPopups | V2 更全面 |
| 按钮优先级 | 取消 → 关闭 → 确定(关二确) | — | V2 有借鉴价值 |
| URL 跳转保护 | 检测并停止清理 | — | V2 有借鉴价值 |
| 弹窗统计 | PopupStats 追踪 | — | 可选 |

**结论**：
- **不建议删除 V3 现有弹窗清理策略**
- V2 的 `clickSmartButton` 优先级策略（关二确 vs 操作二确区分）值得借鉴，但非紧急
- V2 的 URL 跳转保护可在 V3 后续增强

### 4.3 最关键的三个建议

1. **Sign 日期填写必须迁移到 V2 方案**（§2.2）：V3 当前的直接 fill 页面 input 方式不可靠，V2 的面板 fill + 点击"确定"是经过实战验证的稳定方案。

2. **PageSize 和派件员选择考虑 V2 的 evaluate click 降级**（§2.3、§2.4）：V2 在后台标签页场景下使用 evaluate DOM click 绕过 CSS 动画问题，建议 V3 在 Playwright click 失败时降级使用。

3. **所有步骤添加重试机制**：V2 的 MAX_RETRIES=3 + Escape + popup guard 模式应在 V3 中普及（日期、派件员、pageSize 三项当前无重试）。
